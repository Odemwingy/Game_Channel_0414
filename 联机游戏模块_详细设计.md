# IFE Wi-Fi 游戏频道 — 联机游戏模块详细设计

## 1. 架构理念：通用状态机引擎 + 游戏逻辑插件

传统的联机游戏开发中，通常采取“一款游戏一个服务端”的架构（如斗地主 Server、五子棋 Server 独立部署）。但在机载 IFE 环境中，并发受限、资源有限且网络处于局域网隔离状态。为此，我们设计了一套**单进程通用引擎**的架构。

本模块的核心在于将“联机共性逻辑”与“具体游戏规则”解耦分离：
- **联机共性逻辑（Engine）**：负责房间管理、席位分配、玩家加入/离开、断线重线、消息路由转发、回合超时定时机制。由系统核心层统一实现一套即可。
- **具体游戏规则（Plugin）**：负责初始发牌/棋局初始化、用户动作（Action）合法性校验、内部状态（State）流转、终局计算。以独立插件（Plugin）形式开发并挂载到引擎上。

### 2.1 WebSocket Gateway
- **鉴权管理**：基于 Token 或 Device Fingerprint 的连接鉴权。
- **连接保持**：维护全双工长连接，处理底层的 Ping/Pong 心跳包检测断线。
- **路由分发**：解析上行数据包，将协议包路由给下游相关的房间实例。

### 2.2 GameRoom Engine (房间状态机引擎)
每一局创建的联机游戏均为一个独立的 `Room` 实例，引擎负责：
- **生命周期**：房间状态机维护（创建、等待加入、对战中、结算、解散/空置回收）。
- **席位系统**：参与者座位索引分配，区分参战玩家与旁观者视角。
- **事件总线**：捕捉并透传客户端提交的 `Action`，触发对应游戏 Plugin 执行处理。
- **状态广播**：获取 Plugin 更新后的内部状态，并向房间内的在线玩家下行广播更新指令。

### 2.3 GameLogic Plugin (游戏逻辑规则规范)
所有可联机游戏，都需要实现如下 TypeScript 标准接口并在系统启动时注册：

```typescript
export interface GameLogicPlugin<State, Action, PlayerView> {
  // 基础元属性
  readonly gameId: string;
  readonly minPlayers: number;
  readonly maxPlayers: number;

  /**
   * 游戏正式开局，初始化全局完整内部状态
   */
  onGameStart(players: string[]): State; 

  /**
   * 处理玩家动作，驱动状态流转
   */
  onPlayerAction(state: State, playerId: string, action: Action): {
    valid: boolean;       // 动作是否合法
    newState?: State;     // 动作执行后的新内部状态
    events?: any[];       // 产生的附加事件（如发牌动画指令）
    error?: string;       // 不合法时的错误提示信息
  };
  
  /**
   * 视角收敛方法 (关键防作弊机制)
   * 引擎调用此方法将“上帝视角的完整 State”裁剪为当前 playerId 可见的视图
   */
  getPlayerView(state: State, playerId: string): PlayerView;
  
  /**
   * 终局判定
   */
  isGameOver(state: State): { 
      isOver: boolean, 
      winners?: string[], 
      scores?: Record<string, number> 
  };
}
```

## 3. 核心机制：服务端权威 vs 服务端校验

根据游戏特性，Plugin 实现天然支持两类架构设计，一套底层即可兼容：

### 3.1 服务端权威模型 (Server-Authoritative)
**适用场景**：存在极高隐藏信息或概率发牌的游戏（如斗地主、德州扑克、麻将）。
**运行机制**：
- 洗牌池以及所有玩家的真实手牌均绝对保存在 Server 内存中（`State`）。
- Server 广播通过 `getPlayerView(state, myId)` 进行数据掩膜，不发送其余玩家未亮出的手牌数据。
- 客户端不具备任何决策逻辑，UI 为纯渲染层，接收发牌视图并通过 `Action` 申请操作。

### 3.2 服务端校验模型 (Server-Validation)
**适用场景**：完全明牌、无隐藏信息博弈的游戏（如五子棋、象棋、跳棋）。
**运行机制**：
- 棋盘数据对全部玩家透明，`getPlayerView` 基本直接返回完整的全局 `State`。
- 客户端允许先行响应动画/落子反馈（以达到极低的视觉延迟），同时后台发送 `Action` 执行异步同步。
- 若客户端提交作弊 `Action`，Plugin 的 `onPlayerAction` 返回 `valid:false`，服务端下发警告及当前正确的锚点状态，将客户端棋盘状态进行强制拉回复归（回滚）。

## 4. 联机通信协议设计

前后端通过通用 JSON 进行报文交互：

### 4.1 客户端上行 (Client -> Server)
| Topic | Payload (JSON) | 描述 |
|-------|-------------|------|
| `room:join` | `{ roomId: "xxxx" }` | 申请加入已创建的指定房间 |
| `room:leave` | `{}` | 离开/退出大厅 |
| `game:ready` | `{}` | 举手动作：玩家准备完毕可以开始 |
| `game:action` | `{ type: "PLAY_CARD", payload: { cards: ["3♠"] } }` | 局内具体游戏动作协议（透传给 Plugin 解析） |

### 4.2 服务端下行 (Server -> Client)
| Topic | Payload (JSON) | 描述 |
|-------|-------------|------|
| `room:sync` | `{ players: [{id:"A",status:"..."}, ...], state: "WAITING" }` | 大厅人员变动广播同步 |
| `game:start` | `{ view: {...} }` | 游戏就绪开局，下发通过 getPlayerView 裁剪的初始视图 |
| `game:update`| `{ view: {...}, lastAction: {...} }` | 回合变动，下发更新后的视图及触发此更新的上一个动作详情 |
| `game:error` | `{ message: "不符合出牌规则" }` | 动作校验被驳回反馈 |
| `game:over`  | `{ winners: ["A", "B"], payload: {...} }` | 单局结束及最终计分算账广播 |

## 5. 断流阻抗与断线重连 (Reconnection)

机载 Wi-Fi 常因设备休眠或切换应用而断流导致长连接中断。必须提供友好的会话恢复能力。

1. **会话保留与掉线判定 (Session Keep-Alive)**
   - 遇到连接中断（如未按时收到心跳包、Socket EOF），Gateway 不视为“主动退出”。
   - 触发离线中断事件，Engine 判断玩家状态修改为 `OFFLINE`，房间内广播给其他席位：“玩家A网络出现波动”，并倒计时 120 秒。
2. **托管机制 (AI/Bot Hosting)**
   - 当轮到 `OFFLINE` 状态玩家的回合且超时，Engine 会自动根据 Plugin 默认配置下发 fallback action（如：斗地主超时自动“不出”或者托管 AI 接管）。
3. **极速恢复 (Fast Recovery)**
   - 玩家重连并经 Gateway 拿到旧 Session 后下发 `room:reconnect`。
   - Engine 结束离线倒计时，恢复其为 `ONLINE`。
   - Engine 拉取 Plugin 当前最新的 `State`，运行 `getPlayerView(state, targetId)` 投递 `game:start/sync_full` 进行视窗**全量覆盖强刷新**，客户端 UI 无痕重绘出当前赛局现状。
